Automator 是 macOS 平台上一个自动化工具，它提供了一系列基础动作模块，像搭积木一样，用户通过选择拖动这些模块，就可以自动运行一连串的动作，即工作流。
由于 Automator 的制作流程基本是在图形化界面上进行，所以即使没有编程基础的用户也可以轻松制作出适合自己的工作流。熟悉少数派的 macOS 用户一定对 Automator 并不陌生，很多作者都分享过利用 Automator 制作的工具。
在 Automator 中新建动作流时，需要选择工作流的类型：
  • Workflow（工作流程）、
  • Application（应用程序）、
  • Service（服务）、
  • …

在 Automator 中创建的 Service 可以直接用 LaunchBar 调用。使用方法与系统提供的服务以及各类动作的使用方法一致，本章会列举一些简单实例。但是由于 LaunchBar 对 Service 工作流的支持并不稳定，本章亦在后面提出了一种解决这个问题的方法。

把工作流制作成动作

非常遗憾的是，LaunchBar 对 Automator 创建的工作流 Service 支持并不是特别稳定。如果对创建完成的工作流 Service进行重命名等操作，可能会导致 LaunchBar 无法正常触发工作流。
把工作流打包制作成动作，是解决这个问题的一个好办法。拿上一节中介绍的 Convert Images to PDF Document 工作流为例，我们要把它制成一个动作，步骤如下：

1. 制作一个与 Service 结构相同的 Application

接着创建同上例中相同的结构，并命名为「Pic-PDF_Converter.app」。

sys.argv[1:] 即表示输入的文件名称的列表，所以直接将它与前面的列表用加号连接即建立了我们想要的完整列表。

输入项与工作流类型的选择

上一节中讲述的例子，是把文件作为输入传递给工作流 Application。其扩展名为 .app。在使用它时，我们是用它来打开文件，所以使用的 Shell 命令为：
open -a Pic-PDF_Converter.app.app 图片名称列表
由于工作流 Application 只能接受文件作为输入，当我们要创建的工作流接受的是文本时，则不能把这个工作流创建成 Application，而应该创建为 Workflow：

创建完成的文件的扩展名为 .workflow，此时我们在 Terminal 中调用它的 Shell 语句为：
automator -i "文本内容" 工作流名称.workflow
此 .workflow 文件也应添加到这个动作的 Scripts 文件夹中。

如果要制作的工作流没有任何的输入，则既可以把它创建为 .app 文件，亦可以把它创建为 .workflow 文件。
由此做一下总结，在把工作流创建成动作的时候，如果你的工作流没有输入，则：
  • 可在 Scripts 文件夹中创建 .app（Application） 文件，使用的 Shell 语句为：工作流名称.app
  • 或在 Scripts 文件夹中创建 .workflow（Workflow） 文件，使用的 Shell 语句为：automator 工作流名称.workflow
如果你的工作流有输入，则：
  • 有文件输入，需在 Scripts 文件夹中创建 .app（Application）文件，使用的 Shell 语句为：open -a 工作流名称.app 文件名称
  • 有文本输入，需在 Scripts 文件夹中创建 .workflow （Workflow）文件，要使用的 Shell 语句为：automator -i "文本内容" 工作流名称.workflow

🗣闲话

从上面的内容可以看出，虽然 LaunchBar 支持的所谓的工作流，但仅仅是借用系统工具 Automator 创建的工作流，而并非自己的一套工作流制作工具，而且对于 Automator 创建的工作流的支持也并不稳定。打包成一个动作的过程，本质上是把 Automator 工作流本身作为一个脚本，再用动作的主脚本进行调用。
此时这里工作流的概念已与传统的工作流（如 Alfred 所提供的工作流）不相符，只能算得上一种「伪工作流」。
如本章开头所述，少数派上有不少的 Automator 工作流实例，你可以尝试用本章介绍的方法把这些工作流做成 LaunchBar 动作。
如果你没有编程经验，对上述关于代码的讲解仍存在疑问，欢迎在评论区提问。在接下来的答疑汇总中，将详细介绍一个好学易懂的套路，来制作调用命令行工具的 Python 脚本，以帮助没有编程经验的读者也能快速上手，制作出适合自己的 LaunchBar 动作。
